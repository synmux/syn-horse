## Wrangler v4.92.0 Best Practices

## Best Practices

- Keep `compatibility_date` current on all projects — set it to today's date on new projects, then update periodically to unlock new APIs and bug fixes without requiring code changes [source](./.skilld/docs/workers/best-practices/workers-best-practices.md#keep-your-compatibility-date-current)

- Enable `nodejs_compat` flag in configuration — many npm libraries depend on Node.js built-in modules (crypto, buffer, stream), and enabling this flag prevents cryptic import errors that only appear at runtime [source](./.skilld/docs/workers/best-practices/workers-best-practices.md#enable-nodejs_compat)

- Run `wrangler types` after every binding change — never hand-write the `Env` interface as it drifts from actual configuration; let Wrangler generate type definitions that match your config exactly to catch mismatches at compile time [source](./.skilld/docs/workers/best-practices/workers-best-practices.md#generate-binding-types-with-wrangler-types)

- Store secrets with `wrangler secret put`, not in source — API keys, tokens, and credentials must never appear in wrangler.jsonc or code; use `.env` files (in `.gitignore`) for local development and `wrangler secret` commands for all environments [source](./.skilld/docs/workers/best-practices/workers-best-practices.md#store-secrets-with-wrangler-secret-not-in-source)

- Configure environments deliberately with explicit bindings per environment — each environment (`production`, `staging`, etc.) is a separate Worker deployment, and bindings are non-inheritable, so declare all bindings and vars in each environment block even if they appear redundant [source](./.skilld/docs/workers/best-practices/workers-best-practices.md#configure-environments-deliberately)

- Use custom domains for origin-only workers, routes for middleware in front of existing origins — custom domains have automatic DNS and SSL setup; routes require a proxied DNS record and are meant to intercept traffic before reaching your origin [source](./.skilld/docs/workers/best-practices/workers-best-practices.md#set-up-custom-domains-or-routes-correctly)

- Stream large request and response bodies with `response.body` or `TransformStream` — never buffer entire payloads with `await response.text()` or `await request.arrayBuffer()` as Workers have a 128 MB memory limit and buffering large payloads crashes the isolate [source](./.skilld/docs/workers/best-practices/workers-best-practices.md#stream-request-and-response-bodies)

- Use `ctx.waitUntil()` for background work after response is sent — pass the context object directly (do not destructure) to perform analytics, cache updates, and non-critical tasks without blocking the response, respecting the 30-second timeout after response [source](./.skilld/docs/workers/best-practices/workers-best-practices.md#use-waituntil-for-work-after-the-response)

- Prefer bindings over REST API calls for Cloudflare services — R2, KV, D1, Queues, and Workflows are available as bindings (in-process, no auth needed, no latency); calling the REST API from a Worker wastes network time and adds unnecessary complexity [source](./.skilld/docs/workers/best-practices/workers-best-practices.md#use-bindings-for-cloudflare-services-not-rest-apis)

- Use Queues for single-step async work and Workflows for multi-step processes — Queues are message brokers for simple background jobs; Workflows are durable execution engines for jobs with dependent steps that need to pause, resume, or retry individual steps [source](./.skilld/docs/workers/best-practices/workers-best-practices.md#use-queues-and-workflows-for-async-and-background-work)

- Use service bindings for Worker-to-Worker communication instead of HTTP calls — service bindings bypass the public internet, avoid network cost, add zero latency, and support type-safe RPC with better error handling than HTTP fetch [source](./.skilld/docs/workers/best-practices/workers-best-practices.md#use-service-bindings-for-worker-to-worker-communication)

- Enable observability (logs and traces) before production and use structured JSON logging — `console.log` and `console.error` with JSON payloads are searchable and filterable; set `head_sampling_rate` to control volume on high-traffic Workers [source](./.skilld/docs/workers/best-practices/workers-best-practices.md#enable-workers-logs-and-traces)

- Never store request-scoped state in global variables — Workers reuse isolates across requests, so module-level variables persist and leak data between requests causing "Cannot perform I/O on behalf of a different request" errors; pass state through function arguments instead [source](./.skilld/docs/workers/best-practices/workers-best-practices.md#do-not-store-request-scoped-state-in-global-scope)

- Always await or pass Promises to `ctx.waitUntil()` — floating promises (not awaited, returned, or passed to waitUntil) silently drop results and swallow errors; enable the `no-floating-promises` lint rule (`@typescript-eslint/no-floating-promises` for ESLint) to catch these at development time [source](./.skilld/docs/workers/best-practices/workers-best-practices.md#always-await-or-waituntil-your-promises)
