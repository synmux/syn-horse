# AGENTS.md

## Project Snapshot

- Cloudflare Worker for notification processing (`wrangler.jsonc` `main: src/index.ts`). It is primarily a **queue consumer** for `syn-horse-notifications` and also exposes a small `fetch` handler serving a single `/ack` HTTP endpoint.
- Runtime bindings currently configured: `KV`, `DB` (D1), `EMAIL` (Cloudflare Email), `AI` (Workers AI).
- Secrets (`wrangler.jsonc` `secrets.required`): `NTFY_TOKEN`, `SELF_TOKEN`, `PUSHOVER_APP_TOKEN`, `PUSHOVER_USER_TOKEN`. Plain vars: `NTFY_SERVER`, `NTFY_TOPIC`.
- Code is TypeScript ESM with explicit `.ts` import extensions and strict typechecking.

- Node version pinned to 24.16.0 (`.node-version`, `.tool-versions`, `mise.toml`).

## Architecture

### Entry and control flow

- `src/index.ts` exports both a `queue` handler (the primary entrypoint, processing each queue message) and a `fetch` handler that delegates to `handleAck` in `src/http.ts`.
- Per-message flow is stage-based, and **all stages are currently active**:
  1. validate payload (`safeParseMessage`)
  2. logging stage (`runLogging`)
  3. rate-limit stage (`runRateLimits`)
  4. AI moderation stage (`runAi`)
  5. delivery stage (`runDelivery`, called with the `"stop"` failure mode)
- Any stage can short-circuit by returning `STOP`; the handler then `ack()`s the message and skips the remaining stages.

### HTTP `/ack` endpoint

- `src/http.ts` (`handleAck`) backs the `fetch` handler and serves a single route, `/ack`; any other path returns `404`.
- It exists for the ntfy notification's "ack" action button (see `src/adapters/ntfy.ts`), which calls back into the Worker.
- Requires two headers, `X-Message-Id` and `X-Self-Token`. The token is checked against `env.SELF_TOKEN` (`401` on mismatch, `400` when a header is missing). `OPTIONS` is answered as a permissive CORS preflight (`204`); a valid request returns `200`.
- The endpoint currently only validates the shared secret — it does not yet persist anything to D1.

### Stage contract

- Stages return `{ kind: "continue" }` or `{ kind: "stop" }` from `src/stages/types.ts` (exported as the `CONTINUE` / `STOP` singletons).
- `stop` means terminal action already taken (drop/record) and pipeline short-circuits.
- Handler ACKs invalid schema messages (no retry), retries only on thrown runtime errors.

### Data model and persistence

- Payload schema in `src/schema.ts` uses `z.strictObject` (unknown fields fail validation).
- Allowed channels are `red | green`; `source` optional and must be IPv4/IPv6/hostname.
- D1 log table created in `migrations/0001_create_log_table.sql` with CHECK constraints for stage/result enums.
- DB helpers in `src/db.ts` (`insertLogRow`, `updateRateLimit`, `updateAi`, `updateDelivery`) update a single row keyed by queue message ID through the pipeline.

### Rate limiting design (active in the handler)

- Limits in `src/constants.ts`: `hour=10`, `day=100`, `lifetime=1000`.
- Counter keys are KV namespaced by `rate-limits:<source|unknown>:<period>`; periods are checked tightest-first (`hour`, `day`, `lifetime`) so the most specific violated window is reported.
- `src/kv.ts` stores `expiresAt` metadata to preserve fixed windows on increment; the `lifetime` window has no TTL.
- Fail-open behavior in `src/stages/rate-limits.ts`: KV errors log `kv_error` and continue.

### AI moderation design (active in the handler)

- `src/stages/ai.ts` calls Workers AI model `@cf/google/gemma-4-26b-a4b-it`.
- Expects structured JSON via `response_format.type = "json_schema"` (schema derived from a Zod schema via `z.toJSONSchema`) and validates the result with Zod.
- Labels `nonsense`/`spam` drop (`STOP`); `none`/`fun` continue. The channel steers borderline calls: `red` errs towards dropping, `green` towards delivering.
- Empty, non-JSON, or schema-invalid model output throws so the handler retries rather than passing bad output through.

### Delivery/adapters

- Adapter interface (`Adapter`) and the `Notification` payload are in `src/types.ts`; the registry is an explicit switch in `src/adapters/index.ts` (`getAdapter`).
- `runDelivery` (`src/stages/delivery.ts`) selects the adapter by **paging channel** via `ADAPTER_BY_CHANNEL`: `green → ntfy`, `red → pushover`, resolving each by name through `getAdapter`.
- It takes a `DeliveryFailureMode` (`"stop"` | `"skip"`): `stop` aborts the chain on failure (a thrown error is re-thrown so the queue retries; a `false` return is terminal `failed` with no retry); `skip` records the failure and tries the next adapter, ending `delivered` if any adapter accepted. The handler currently passes `"stop"`.
- Every attempted adapter is recorded (comma-separated) in the `adapter` column, with failures surfaced in `result_reason`.
- Adapter status: `ntfy` and `pushover` are **real** implementations; `email` is a placeholder returning `true`; `stub` is a no-op success adapter (registered, but no longer in the active channel routing); `twilio` (`src/adapters/twilio.ts`) is a placeholder and is **not** yet registered in `getAdapter`.

## Core Principles

Write code that is **accessible, performant, type-safe, and maintainable**. Focus on clarity and explicit intent over brevity.

### Type Safety & Explicitness

- Use explicit types for function parameters and return values when they enhance clarity
- Prefer `unknown` over `any` when the type is genuinely unknown
- Use const assertions (`as const`) for immutable values and literal types
- Leverage TypeScript's type narrowing instead of type assertions
- Use meaningful variable names instead of magic numbers - extract constants with descriptive names

### Modern JavaScript/TypeScript

- Use arrow functions for callbacks and short functions
- Prefer `for...of` loops over `.forEach()` and indexed `for` loops
- Use optional chaining (`?.`) and nullish coalescing (`??`) for safer property access
- Prefer template literals over string concatenation
- Use destructuring for object and array assignments
- Use `const` by default, `let` only when reassignment is needed, never `var`

### Async & Promises

- Always `await` promises in async functions - don't forget to use the return value
- Use `async/await` syntax instead of promise chains for better readability
- Handle errors appropriately in async code with try-catch blocks
- Don't use async functions as Promise executors

### React & JSX

- Use function components over class components
- Call hooks at the top level only, never conditionally
- Specify all dependencies in hook dependency arrays correctly
- Use the `key` prop for elements in iterables (prefer unique IDs over array indices)
- Nest children between opening and closing tags instead of passing as props
- Don't define components inside other components
- Use semantic HTML and ARIA attributes for accessibility:
  - Provide meaningful alt text for images
  - Use proper heading hierarchy
  - Add labels for form inputs
  - Include keyboard event handlers alongside mouse events
  - Use semantic elements (`<button>`, `<nav>`, etc.) instead of divs with roles

### Error Handling & Debugging

- Remove `console.log`, `debugger`, and `alert` statements from production code
- Throw `Error` objects with descriptive messages, not strings or other values
- Use `try-catch` blocks meaningfully - don't catch errors just to rethrow them
- Prefer early returns over nested conditionals for error cases

### Code Organization

- Keep functions focused and under reasonable cognitive complexity limits
- Extract complex conditions into well-named boolean variables
- Use early returns to reduce nesting
- Prefer simple conditionals over nested ternary operators
- Group related code together and separate concerns

### Security

- Add `rel="noopener"` when using `target="_blank"` on links
- Avoid `dangerouslySetInnerHTML` unless absolutely necessary
- Don't use `eval()` or assign directly to `document.cookie`
- Validate and sanitize user input

### Performance

- Avoid spread syntax in accumulators within loops
- Use top-level regex literals instead of creating them in loops
- Prefer specific imports over namespace imports
- Avoid barrel files (index files that re-export everything)
- Use proper image components (e.g., Next.js `<Image>`) over `<img>` tags

### Framework-Specific Guidance

**Next.js:**

- Use Next.js `<Image>` component for images
- Use `next/head` or App Router metadata API for head elements
- Use Server Components for async data fetching instead of async Client Components

**React 19+:**

- Use ref as a prop instead of `React.forwardRef`

**Solid/Svelte/Vue/Qwik:**

- Use `class` and `for` attributes (not `className` or `htmlFor`)

## Testing

- Write assertions inside `it()` or `test()` blocks
- Avoid done callbacks in async tests - use async/await instead
- Don't use `.only` or `.skip` in committed code
- Keep test suites reasonably flat - avoid excessive `describe` nesting

## When Biome Can't Help

Biome's linter will catch most issues automatically. Focus your attention on:

1. **Business logic correctness** - Biome can't validate your algorithms
2. **Meaningful naming** - Use descriptive names for functions, variables, and types
3. **Architecture decisions** - Component structure, data flow, and API design
4. **Edge cases** - Handle boundary conditions and error states
5. **User experience** - Accessibility, performance, and usability considerations
6. **Documentation** - Add comments for complex logic, but prefer self-documenting code
