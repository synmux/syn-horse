## Architecture

### Entry and control flow

- `src/index.ts` is a queue handler (no `fetch` handler) that processes each queue message.
- Message flow is stage-based:
  1. validate payload (`safeParseMessage`)
  2. logging stage (`runLogging`)
  3. rate-limit stage (`runRateLimits`)
  4. AI moderation stage (`runAi`)
  5. delivery stage (`runDelivery`)
- **Current behavior:** stages 3 and 4 are commented out in `src/index.ts`; only logging + delivery are active.

### Stage contract

- Stages return `{ kind: "continue" }` or `{ kind: "stop" }` from `src/stages/types.ts`.
- `stop` means terminal action already taken (drop/record) and pipeline short-circuits.
- Handler ACKs invalid schema messages (no retry), retries only on thrown runtime errors.

### Data model and persistence

- Payload schema in `src/schema.ts` uses `z.strictObject` (unknown fields fail validation).
- Allowed channels are `red | green`; `source` optional and must be IPv4/IPv6/hostname.
- D1 log table created in `migrations/0001_create_log_table.sql` with CHECK constraints for stage/result enums.
- DB helpers in `src/db.ts` update a single row keyed by queue message ID through the pipeline.

### Rate limiting design (implemented, currently not wired in handler)

- Limits in `src/constants.ts`: `hour=10`, `day=100`, `lifetime=1000`.
- Counter keys are KV namespaced by `rate-limits:<source|unknown>:<period>`.
- `src/kv.ts` stores `expiresAt` metadata to preserve fixed windows on increment.
- Fail-open behavior in `src/stages/rate-limits.ts`: KV errors log `kv_error` and continue.

### AI moderation design (implemented, currently not wired in handler)

- `src/stages/ai.ts` calls Workers AI model `@cf/google/gemma-4-26b-a4b-it`.
- Expects structured JSON via `response_format` + Zod validation.
- Labels `nonsense`/`spam` drop; `none`/`fun` continue.

### Delivery/adapters

- Adapter interface in `src/types.ts`; registry is explicit switch in `src/adapters/index.ts`.
- `runDelivery` is currently hardcoded to `getAdapter("stub")`.
- `email` adapter is a placeholder returning `true`; `ntfy` and `pushover` are real implementations.
- `stub` is a no-op success adapter used by current pipeline.
